# 优化待办清单

## 开头说明

这份清单面向当前 `SoftRenderer` 仓库的后续优化实施，目标不是做研究综述，而是给自己或协作者一份可以直接排期和落地的执行文档。

排序原则按“改动小且收益明确”到“架构级重构”展开，优先把低风险、高确定性的事项放在前面，再逐步进入需要重写热路径、调整数据流或引入并行结构的大项。

难度和收益都只使用定性等级，不写未经 profile 支撑的精确倍数或工时：

- 难度关注代码侵入性、联动范围和调试成本。
- 收益关注对当前真实热路径的影响，而不是理想化上限。
- 每一项都会补一句评估依据，避免只有标签没有判断理由。

## 评估标准

- 实现难度：`低 / 中低 / 中 / 中高 / 高`
- 预期收益：`低 / 中 / 高 / 很高`

判断口径如下：

- `低`：局部修改、影响面窄、容易验证正确性。
- `中低`：仍以局部改动为主，但需要跨 2 到 3 个模块协调。
- `中`：会改变阶段顺序、函数接口或中间数据流，需要额外回归验证。
- `中高`：涉及热路径重写、算法替换或多个阶段协同调整。
- `高`：带来新的执行模型或新的数据布局，开发和调试成本都显著上升。

## 优化清单

### 1. 补 profiling / benchmark 基线

- 问题：
  当前项目缺少稳定的性能基线，`main.cpp` 里的 workload 也很小，后续优化容易陷入“改了很多但不知道值不值”的状态。
- 实现思路：
  在渲染主路径里为 `vertex_shader_stage`、clip、raster、fragment、depth、present 增加分段计时；再准备最小 benchmark 场景，至少覆盖全屏纹理面片、高 overdraw 三角形和基础透明混合。
- 实现难度：`低`
  评估依据：主要是加观测点和整理测试入口，不需要先改渲染算法。
- 预期收益：`高`
  评估依据：它不直接提速，但能避免后续优化方向跑偏，是所有大改造的前置收益。
- 前置条件 / 风险：
  需要先约定统一的统计口径，例如记录 frame time、triangles/s、pixels/s、allocs/frame；否则不同优化前后的数据不可比。
- 验收信号：
  能稳定输出分阶段耗时，并在同一场景下多次运行得到趋势一致的基线数据。

### 2. 消除热路径里的低成本拷贝和容器扩容

- 问题：
  当前代码在热路径里有一些容易忽视但持续存在的杂费，例如 `VertexArrayObject::get_binding_map()` 按值返回、多个临时 `vector` 默认从空开始扩容。
- 实现思路：
  先把 `get_binding_map()` 改为返回只读引用或通过外部缓存复用；对 `vsOutputs`、`clipOutputs`、裁剪临时结果等容器补 `reserve()`；顺手减少不必要的中间复制。
- 实现难度：`低`
  评估依据：改动集中在容器和接口细节层面，逻辑行为基本不变。
- 预期收益：`中`
  评估依据：这类改动不会改变算法上限，但能稳定减少固定 CPU 开销和内存抖动。
- 前置条件 / 风险：
  如果把返回值改成引用，需要确认生命周期和只读约束，避免引入悬垂引用或误写共享状态。
- 验收信号：
  行为结果不变，热路径里的临时分配次数下降，接口调用链更少发生复制。

### 3. 减少顶点属性读取的 `map` 查找和 `memcpy` 开销

- 问题：
  `Shader::get_vector()` 每次读取属性都要做 binding 查找、VBO 查找和一次 `memcpy`；shader 又会重复取 position、color、uv，导致顶点抓取路径偏重。
- 实现思路：
  把 attribute binding 预解析成更热路径友好的描述结构；按 shader 使用模式缓存 position/color/uv 的偏移与 stride；让顶点阶段尽量少走“通用查找”而更多走“预绑定读取”。
- 实现难度：`中低`
  评估依据：需要跨 `VAO`、`Shader`、具体 shader 实现联动，但还不涉及主渲染阶段重排。
- 预期收益：`中`
  评估依据：当前 demo 顶点量不大时收益有限，但在三角形数量增多后会成为稳定收益项。
- 前置条件 / 风险：
  要保证优化后仍兼容当前 attribute 配置方式，避免为了加速单一路径而把接口写死得过早。
- 验收信号：
  顶点阶段读取逻辑更直达，单顶点属性抓取的查找次数明显减少，图像结果与当前一致。

### 4. 把 opaque 路径调整为先 depth test 再 fragment shading

- 问题：
  当前主路径里是先跑 `fragment_shader()`，再做 `depth_test()`，这意味着被遮挡片元也会去采样纹理、计算颜色。
- 实现思路：
  把 draw 流程拆成 opaque 和 alpha-blend 两条路径；opaque 路径先确定像素坐标和深度，通过深度测试后再执行 fragment shader，透明路径继续保留按需求处理的顺序。
- 实现难度：`中`
  评估依据：这会改变阶段顺序和 draw 行为，需要明确哪些材质能走 early-z，哪些不能。
- 预期收益：`高`
  评估依据：只要 overdraw 存在，这类“先挡掉再着色”的优化就会立刻减少片元浪费。
- 前置条件 / 风险：
  需要先定义 opaque 与透明 pass 的区分规则，否则容易把有透明需求的路径错误地提前裁掉。
- 验收信号：
  opaque 场景结果不变，被遮挡片元的 fragment 执行次数减少，高 overdraw 场景耗时明显下降。

### 5. 去掉 `rasterOutputs` 全量片元缓存，改为流式消费

- 问题：
  当前 `Raster::rasterize()` 会先把所有覆盖片元展开到 `rasterOutputs`，再统一恢复透视、执行 fragment 和写回 framebuffer，中间片元缓存既耗内存也耗带宽。
- 实现思路：
  把 rasterizer 改成 callback / consumer 风格：每生成一个片元就立即进入透视恢复、深度测试、着色和写回，而不是先完整收集再统一消费。
- 实现难度：`中`
  评估依据：需要改主路径数据流和 rasterizer 接口，但总体仍沿用现有阶段划分。
- 预期收益：`很高`
  评估依据：当前版本的中间片元向量直接位于热路径中心，去掉后能同时降低分配、写入和读回成本。
- 前置条件 / 风险：
  需要先确认 line 和 triangle 两条 raster 路径都能接入统一 consumer 接口，避免新接口只服务单一图元类型。
- 验收信号：
  主路径中不再存在按整批片元缓存的 `vector`，全屏和高覆盖场景下内存流量明显下降。

### 6. 重写三角形光栅核心为“一次 setup + 增量递推”

- 问题：
  当前三角形光栅化对每个候选像素都重复计算 cross、inside test 和一部分 setup 逻辑，CPU 在大量重复做本可预处理的工作。
- 实现思路：
  引入 triangle setup 结构，预先计算 bbox、边函数参数和覆盖判断所需常量；逐像素阶段改用增量 edge function 递推，而不是每次从头算三组向量叉积。
- 实现难度：`中高`
  评估依据：这是对核心热循环的算法级改写，正确性和边界规则都需要重新核对。
- 预期收益：`很高`
  评估依据：当前 rasterizer 的像素循环是最显眼的热点之一，替换为增量递推后上限提升最直接。
- 前置条件 / 风险：
  需要明确像素中心规则、边界包含规则和退化三角形处理方式，不然容易出现裂缝或覆盖差异。
- 验收信号：
  大三角形和高 fill-rate 场景耗时显著下降，同时边缘覆盖结果保持稳定、没有新增裂缝。

### 7. 把透视相关插值改为平面方程递推

- 问题：
  当前 `interpolant_triangle()` 在每个像素上都重新通过面积比重建权重，再分别插值 `z`、`inv_w`、`color`、`uv`，计算模式偏重。
- 实现思路：
  在 triangle setup 阶段预计算 `z`、`invW`、`uOverW`、`vOverW` 等平面方程系数；光栅阶段按像素递推这些值，片元阶段再恢复真实属性。
- 实现难度：`中高`
  评估依据：它与第 6 项强相关，属于插值模型的整体替换，不是单点修补。
- 预期收益：`高`
  评估依据：它能继续压低每像素计算量，并让透视恢复和纹理采样的数据来源更稳定。
- 前置条件 / 风险：
  最适合作为第 6 项之后的继续优化；若在旧 raster 结构上单独做，收益会被当前重复覆盖测试抵消一部分。
- 验收信号：
  透视正确性保持不变，插值路径从“面积重算”切换为“系数递推”，片元阶段算术密度下降。

### 8. 纹理采样路径专用化

- 问题：
  当前 `Texture::get_color()` 在片元热循环里同时承担 wrap 处理、nearest / linear 分支和多次 `round / floor / ceil` 运算，动态分支较多。
- 实现思路：
  按 filter 和 wrap 模式拆分更专用的采样路径，尽量把模式判断移到材质或纹理状态准备阶段；保留当前功能，但减少采样时的动态判断和重复数学运算。
- 实现难度：`中高`
  评估依据：要改纹理接口和调用组织方式，同时保持采样结果与当前一致并不轻松。
- 预期收益：`高`
  评估依据：在纹理片元占比高的场景里，减少采样分支通常能带来持续收益。
- 前置条件 / 风险：
  需要小心 repeat、mirror、nearest、linear 组合路径的行为一致性，避免不同专用路径之间结果飘移。
- 验收信号：
  纹理采样逻辑从“单函数多分支”转成“少分支专用路径”，纹理密集场景耗时下降且图像一致。

### 9. 引入 tile 化结构，但先保持单线程

- 问题：
  当前渲染流程以全屏共享 framebuffer 为中心组织，虽然单线程能跑，但对 cache locality 和后续并行化都不友好。
- 实现思路：
  先引入 tile bins、tile scratch color/depth 和 tile flush 结构，让图元先按 tile 组织，再在单线程下验证 tile 模式的正确性和收益。
- 实现难度：`高`
  评估依据：这已经不是局部调优，而是在给整个渲染后端换组织方式。
- 预期收益：`高`
  评估依据：单线程阶段未必立刻翻倍提速，但它是后续并行扩展和 cache 友好访问的关键基础。
- 前置条件 / 风险：
  最好在第 5 到 8 项完成后推进，否则旧的中间数据流会干扰 tile 结构设计。
- 验收信号：
  单线程 tile 路径能够稳定产出与旧路径一致的图像，并开始展现更好的局部性和更清晰的后端结构。

### 10. 基于 tile owner 的多线程渲染

- 问题：
  当前 `GPU` 和 `Application` 设计都偏共享可变状态，直接把现有代码硬拆成多线程，很容易遇到竞态、伪共享和同步争用。
- 实现思路：
  基于 tile owner 模型做并行化：每个 tile 只由固定 worker 独占写入，任务调度围绕 tile 级工作展开，避免多个线程同时写同一像素。
- 实现难度：`高`
  评估依据：它会引入新的执行模型、同步边界和调试维度，是典型的架构级改动。
- 预期收益：`很高`
  评估依据：一旦单线程后端结构合理，多核 CPU 的扩展收益通常会集中体现在这一步。
- 前置条件 / 风险：
  必须先有单线程 tile 路径和稳定 baseline；否则多线程问题会和基础算法问题叠在一起，极难定位。
- 验收信号：
  在 2/4/8 线程下能看到基本单调的性能提升，同时图像结果与单线程版本一致。

### 11. SIMD / AoSoA / 顶点与纹理布局压缩

- 问题：
  当前数据布局更偏教学实现，未围绕 SIMD 读取和批量处理优化；顶点颜色、UV 和纹理布局也没有针对 CPU 向量化做准备。
- 实现思路：
  在算法和并行结构稳定后，进一步做 AoSoA 或相近的数据布局重排；将 raster / 插值路径改成按 4 或 8 像素批处理；评估颜色和 UV 的压缩表示以及纹理块状布局。
- 实现难度：`高`
  评估依据：它既依赖前面几项的结构铺垫，也会显著提高实现和调试复杂度。
- 预期收益：`高`
  评估依据：在合适的数据布局和热路径结构上，SIMD 常常是最后一层峰值性能放大器。
- 前置条件 / 风险：
  不适合过早推进；如果在旧的逐像素、重分支、重缓存路径上强上 SIMD，维护成本高而收益不稳。
- 验收信号：
  向量化路径能稳定通过正确性回归，在目标 ISA 上比标量实现有可重复的性能优势。

## 收尾建议

推荐执行顺序如下：

1. 先补 profiling / benchmark 基线，建立统一比较口径。
2. 先做热路径顺序修正，包括低成本拷贝清理、顶点抓取优化和 opaque early-z。
3. 紧接着去掉 `rasterOutputs` 全量片元缓存，改成流式消费。
4. 然后重写三角形 raster 和透视插值核心。
5. 在单线程热路径稳定后，再推进纹理采样专用化、tile 化结构。
6. 最后才上多线程和 SIMD。

一句结论：

当前项目最值得优先投入的，不是立刻并行化，而是先把主渲染热路径从“中间片元缓存 + 重复像素计算 + 先着色后测深度”改成更直接、更节省带宽的结构。

补充说明：

这份清单里的收益判断基于当前代码路径和典型 CPU 软件光栅化经验，不是目标机器上的最终实测结论；真正排期前，仍应先补 baseline 和 profiling 数据。
